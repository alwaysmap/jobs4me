# Board Generator

Generate a self-contained `Kanban/index.html` file from `tracker.yaml` that the user can open in any browser.

Board rebuilds are fast (~1 second) — the script reads tracker data, injects it into the HTML template, and writes the file. No LLM processing is involved. All skills should rebuild the board after any state change. The board auto-rebuilds on every tracker mutation by default.

## How to Build

**Shell setup:** before running tracker commands, `export JFM_DIR='<workspace path>'` (single quotes). Then omit `--dir` from commands. See `search/references/data-safety.md`.

The board is generated by a Node script — no LLM reasoning needed. Use the script for all board rebuilds:

```bash
node ${CLAUDE_PLUGIN_ROOT}/scripts/tracker.js build-board
```

This reads `tracker.yaml`, `profile.yaml`, `archetypes.yaml`, and `filters.yaml`, reads all markdown documents (JDs, company overviews, interview prep), injects everything into the HTML template, and writes `Kanban/index.html`. It handles all the placeholder replacement, JSON escaping, and file-existence checks.

### Auto-rebuild on mutations

All mutating commands auto-rebuild the board by default. Pass `--no-board` only during batch operations where you want to skip intermediate rebuilds and do one final `build-board` at the end.

```bash
node ${CLAUDE_PLUGIN_ROOT}/scripts/tracker.js stage --id acme-vp-eng --stage interviewing
```

If the user has `Kanban/index.html` open in their browser, they just refresh to see changes.

## Board Features

The generated board is **read-only** — it displays the current state of the tracker but does not write changes back. All state changes happen through Claude (via `/update`, `/assess`, or conversation).

**Document viewer**: The board embeds markdown document contents (JDs, company overviews, interview prep) at build time. Clicking a document link opens a slide-out panel that renders the markdown as formatted HTML using marked.js. No external files needed — everything is self-contained in the board.

Features:
- **Kanban columns**: Suggested, Maybe, Applied, Interviewing, Rejected, Closed
- **Cards**: Company name, role, role type tag, last updated date, notes, recommendation hint
- **Detail popup**: Click a card to see full agent summary, dates, links, and notes
- **Document flyout**: Click JD / Company Overview / Interview Prep links to read rendered docs in a slide-out panel
- **Search/filter**: Type to filter cards by company or role name
- **Config panel**: Toggle to see role types, search sources, target/watch companies, skip list, decline patterns, and preferences
- **Privacy mode**: Toggle to blur sensitive information (for screen sharing) — applies to cards, detail popups, and config panel
- **Declined view**: Toggle to see all declined roles in a grid, grouped by month
- **Responsive**: Works on desktop and mobile

The board does NOT include:
- Drag and drop (changes must go through Claude)
- Save functionality (read-only)
- Briefs panel (briefs are viewed via `/brief` command)
- Interested/Decline buttons (use `/decline` command instead)

## When to Regenerate

The board auto-rebuilds on every tracker mutation. There is no need to check freshness — just let the script run. It's fast enough to run on every change.

## When to Offer the Board

Always tell the user to open or refresh the board after any operation that changes their pipeline — search sweeps, stage changes, declines, review sessions. The board is always current because it auto-rebuilds on every mutation. The user should never need to run `/board` manually unless they want to force a rebuild after editing YAML files by hand.

**IMPORTANT: How to reference the board in messages to the user.**

Always use this exact phrasing with backtick code formatting. NEVER turn it into a markdown link — the file path is not a URL.

Correct: `Your board is updated — refresh Kanban/index.html to see the changes.`

WRONG: `[your Kanban board](some/path/Kanban/index.html)` — NEVER do this. The path is not a clickable URL. Just mention the filename in plain text or backticks.

## Data Format

The template expects three JSON blobs injected at placeholders:

### `__TRACKER_DATA__`

```json
[
  {
    "id": "company-role-kebab",
    "company": "Company Name",
    "role": "Role Title",
    "url": "https://...",
    "stage": "suggested",
    "archetype": "tpm_leadership",
    "last_updated": "2026-03-27",
    "agent_summary": "### Recommendation\n**Strong** — ...",
    "notes": "Short note",
    "decision": { "proceed": "yes", "reason": "Good fit" },
    "dates": { "identified": "2026-03-27", "applied": "2026-03-28" },
    "has_jd": true,
    "has_overview": true,
    "has_prep": false
  }
]
```

### `__CONFIG_DATA__`

Built from `profile.yaml`, `archetypes.yaml`, and `filters.yaml`.

### `__DOCUMENT_DATA__`

A map of document keys → markdown content. JD and prep docs are role-specific (keyed by app ID), while overviews are company-level (keyed by company name):

```json
{
  "acme-corp-vp-engineering::jd": "# VP Engineering\n\nAcme is looking for...",
  "Acme Corp::overview": "# Acme Corp — Company Overview\n\n...",
  "acme-corp-vp-engineering::prep": "# Acme Corp — Interview Prep\n\n..."
}
```

**CRITICAL**: All three JSON blobs are inside `<script type="application/json">` tags. If any JSON contains the literal string `</script>`, escape it to `<\/script>` before injection.
